为什么并发很难?

竞争条件

竞争条件是最难以发现的并发 bug 类型之一.

原子性

当某些东西被认为是原子的, 或者具有原子性的时候, 这意味着在它运行的环境中, 它是不可分割的或不可中断的.

并且原子性在不同的环境(应用, 系统, 硬件)也是不同的.

在考虑原子性时, 经常第一件需要做的事就是定义上下文或范围, 然后再考虑这些操作是否是原子性的, 一切都应当遵循这个原则.

这里举了个例子, 反作弊程序认为扫描机器内存是原子操作, 而作弊程序直接利用硬件中断隐藏自己.

术语 不可分割(indivisible), 不可中断(uninterruptible)

如果一个东西是原子的, 那意味着它在并发环境中是安全的. 而大多数语句不是原子的, 更别说函数, 方法和程序了.

通过各种技术使操作保持原子性, 之后确定哪些部分需要原子化, 以及在什么粒度级别上原子化.

内存访问同步

程序中需要独占访问共享资源的部分叫 临界区(critical section)

内存访问同步虽然能手动解决临界区问题, 并且需要编写者做出约定, 访问临界区要手动同步内存. 单他并不会自动解决数据竞争或者逻辑正确性问题, 此外还带来了维护性和性能问题.

访问同步带来的问题是:

  • 临界区是否频繁的进入和退出?
  • 临界区应该有多大?

死锁, 活锁和饥饿

死锁

1971年, Edgar Coffman 列举了死锁条件, 被称为 Coffman 条件(必须全部满足):

  • 相互排斥, 并发进程同时拥有资源的独占权
  • 等待条件, 并发进程必须同时拥有一个资源, 并等待额外的资源
  • 没有抢占, 并发进程拥有的资源只能被该进程释放, 即可满足这个条件
  • 循环等待, 一个并发进程(P1) 必须等待其他并发进程(P2), 这些并发进程同时也在等待进程(P1), 这样便满足了这个最终条件.

活锁

活锁是正在主动执行并发操作的程序, 但是这些操作无法向前推进程序的状态.

活锁是一组被称为"饥饿"的更大问题的子集.

饥饿

饥饿是在任何情况下, 并发进程都无法获得执行工作所需的所有资源.

可以通过记录(metric)来识别饥饿.

平衡临界区的粒度是解决性能问题和饥饿问题的关键.

外部因素也可能会导致饥饿.

Chapter 3, Go 语言并发组件

goroutine

简单来讲, goroutine是一个并发的函数.

goroutine 是协程, 协程是一种非抢占式的简单并发子 goroutine (函数, 闭包或者方法), 它不能被中断(这里应该是指没有设计中断功能, 但他所属的go runtime被中断是完全可能的). 取而代之的是, 协程有多个point, 允许暂停或重新进入.

go runtime 会调度goroutine并在需要时阻塞造成抢占的现象.

这里再次强调了虽然协程可以并发, 但真正执行时并不一定.

调度模式

Go 语言的主机托管机制是 M:N 调度器实现. 将M个绿色线程(runtime自己管理的线程), 映射到N个OS线程. 然后将 rogoutine 运行在绿色线程上.

Go 语言遵循 fork-join 并发模型, fork 指在程序中的任意一节点, 可以将子节点与父节点同时运行. join 指, 这些并发的执行分支将会合并在一起.

默认之运行 go + func 并没有定义 join point, 因此很有可能看不到 fork 出来的 goroutine 正确执行, 为了正确执行, 要么增加竞争条件 (例如sleep), 要么显式的创建 join point:

var wg sync.WaitGroup
sayHello := func() {
    defer wg.Done()
    fmt.Println("hello")
}
wg.Add(1)
go sayHello()
wg.Wait()

goroutine 会修改闭包中的值, 即值是按照引用传递的.

第二个例子, 循环中嵌套 goroutine 执行的匿名函数.

如果闭包引用值超过生命周期, 则会被runtime转移到堆.

如果按值传递 (通过参数), 则会按照执行顺序去执行.

goroutine 足够轻量化

参考 Go 汇编, 栈的扩容的确开销不大.

如何测量 goroutine 内存消耗:

// calculate goroutine mem usage
package main 

import (
    "fmt"
    "sync"
    "runtime"
)

func main() {
    memConsumed := func() uint64 {
        runtime.GC()
        var s runtime.MemStats
        runtime.ReadMemStats(&s)
        return s.Sys
    }

    var c <-chan interface{}
    var wg sync.WaitGroup
    noop := func() {
        wg.Done(); <-c
    }

    const numGoroutines = 1e8
    wg.Add(numGoroutines)
    before := memConsumed()
    for i:= numGoroutines; i > 0; i-- {
        go noop()
    }

    wg.Wait()
    after := memConsumed()

    fmt.Printf("%.3fkb", float64(after-before)/numGoroutines/1000)
}

cond

可以用cond.Wait暂停 goroutine, 并由cond.Signal()激活. 比sleep等更节省CPU时间片.

package main

import (
    "fmt"
    "sync"
    "time"
)


func main(){
    c := sync.NewCond(&sync.Mutex{})
    queue := make([]interface{}, 0, 10)

    removeFromQueue := func(delay time.Duration) {
        time.Sleep(delay)
        c.L.Lock()
        queue = queue[1:]
        fmt.Println("removed from queue")
        c.L.Unlock()
        c.Signal()
        fmt.Println("roroutein runed")
    }

    for i := 0; i < 10; i++{
        c.L.Lock()
        for len(queue) == 2 {
            c.Wait()
        }
        fmt.Println("Adding to queue")
        queue = append(queue, struct{}{})
        go removeFromQueue(1*time.Second)
        c.L.Unlock()
    }
}

这个例子中在 removeFromQueue goroutine 开始时, sleep了1秒, 为了确保for循环中 unlock 后的间隙不至于被 goroutine 抢到锁. 例子写的不是很好.

cond.Broadcast 同理, 只不过能广播. 这里展示了一个类似注册机制的设计, 每次注册启动一个goroutine并wait. 一旦触发 broadcast, 就执行注册的函数.

这里看到了如果整个 cond 逻辑拖得很长代码管理会很烦, 因此最好封装一下或者控制一下范围.

once

跨 goroutine, 每个声明的once实例只执行一次.

只计算调用次数, 而不是内部注册的函数名称. 因此在同一个函数内代码显式调用多次, 也只会执行一次.

如果两个不同的 sync.Once 实例互相调用, 会造成死锁.

池

即 sync.Pool, 通常用于约束创建昂贵的场景, 例如数据库连接. 同时如果负载很大, 还可以预先分配一些实例, 用于缓冲, 提高性能.

Pool 提供了 Get 和 Put 用于取出和放回. New 用来初始化.

说明 Pool 用途的例子:

package main

import (
    "fmt"
    "sync"
)

func main(){
    var numCalcsCreated int
    calcPool := &sync.Pool {
        New: func() interface{} {
            numCalcsCreated += 1
            mem := make([]byte, 1024)
            return &mem
        },
    }

    // init 4kb
    for j := 4; j >0 ; j--{
        calcPool.Put(calcPool.New())

    }


    const numWokers = 1024 * 1024
    var wg sync.WaitGroup
    wg.Add(numWokers)
    for i := numWokers; i > 0; i-- {
        go func() {
            defer wg.Done()
            mem := calcPool.Get().(*[]byte)
            defer calcPool.Put(mem)
        }()
    }

    wg.Wait()
    fmt.Printf("%d calcs were created.", numCalcsCreated)
}

文中这里表达得不好, 实际上有 Pool 控制, Pool 会根据所需的性能去分配数量. 文中实际上分配了8个. 在我的40核心机器上, 同样的代码分配了26-27个.

这里原文示例代码有问题, 应该是 dataSteam = make(...), 而不是 dataSteam := make(...)

channel 可以隐式转换, 可以精确控制channel用途

var reveiveChan <-chan interface{}
var sendChan chan<- interface{}
dataStream := make(chan interface{})

receiveChan = dataStream
sendChan = dataStream

channel是阻塞的, 默认容量为0, 无缓冲. 这意味着写只能在空的时候写入, 读取只能在有内容的时候读取, 其他都会被阻塞. 所以要注意死锁问题.

channel 可以 close, 并且可以通过读取返回来检测:

intStream := make(chan, int)
close(intStream)

integer, ok := <-intStream
fmt.Printf("(%v):%v", ok, integer)

输出是:

(false): 0

这里衍生了读取channel的模式:

intStream := make(chan int)
go func() {
    defer close(intStream)
    for i := 1; i<=5; i++ {
        intStream <- i
    }
}()

for integer := range intStream {
    fmt.Printf("%v ", integer)
}

关闭channel也可以作为给多个goroutine发信号的方式:

begin := make(chan interface{})
var wg sync.WAitGroup
for i := 0; i< 5;i++ {
    wg.Add(1)
    go func(i int) {
        defer wg.Done()
        <-begin
        fmt.Printf("%v has begun\n", i)
    }(i)
}

fmt.Println("unblocking goroutines...")
close(begin)
wg.Wait()

当然也可以创建 buffered channel:

var dataStream chan interface{}
dataStream = make(chan interface{}, 4)

var stdoutBuff bytes.Buffer
defer stdoutBuff.WriteTo(os.Stdout)

intStream := make(chan int, 4)
go func() {
    defer close(intStream)
    defer fmt.Fprintln(&stdoutBuff, "Producer Done.")
    for i := 0; i < 5; i++ {
        fmt.Frintf(&stdoutBuff, "Sending: %d\n", i)
        intStream <- i
    }
}()


for integer := range intStream {
    fmt.Fprintf(&stdoutBuff, "Received %v.\n", integer)
}

nil channel (未初始化的channel), 无论读取, 写入都会造成死锁错误(尽管可能并不是真的死锁), close的话会有关闭nil channel的错误.

这里整理的表格也有错误, nil channel 读写都应该是 panic.

后续例子的核心即, 利用 channel的特性, 将channel的生命周期控制在channel的写入者手里, 并且范围尽可能小. 避免出现问题.

Chapter 4, Go 语言的并发模式

约束

编写并发代码时, 可以有保证安全的方法:

  • 用于共享内存的同步原语 (sync.Mutex等)
  • 通过通信共享内存来进行同步 (channel)

也有隐式并发安全的:

  • 不会发生改变的数据 (函数式编程)
  • 受保护的数据

这里一大段例子讲了明确约束范围的好处, 避免将需要保护的数据暴露给外围从而达成并发安全.

package main
import (
    "bytes"
    "sync"
    "fmt"
)

func main() {
    printData := func(wg *sync.WaitGroup, data []byte) {
        defer wg.Done()
        var buff bytes.Buffer
        for _, b := range data {
            fmt.Fprintf(&buff, "%c", b)
        }
        fmt.Println(buff.String())
    }

    var wg sync.WaitGroup
    wg.Add(2)
    data := []byte("golang")
    go printData(&wg, data[:3])
    go printData(&wg, data[3:])

    wg.Wait()
}

这段示例代码讲的是必须手动切分数据来完成并发, 这不是好的方式.

防止 goroutine 泄漏

goroutine的终止方式有:

  • 生命周期结束
  • fatal error
  • 被终止

goroutine 泄露的例子:

doWork := func(strings <-chan string) <-chan interface{} {
    completed := make(chan interface{})
    go func() {
        defer fmt.Println("doWork exited.")
        defer close(completed)
        for s :== range strings {
            // do sth
            fmt.Println(s)
        }
    }()
    return completed
}

doWork(nil)
// maybe other works in here

fmt.Println("Done.")

好的实践是父goroutine与子goroutine之间实现信号, 方便父goroutine实现控制.

doWork := func(done <-chan interface{}, strings <-chan string) <-chan interface{} {
    terminated := make(chan interface{})
    go func() {
        defer fmt.Println("doWork exited.")
        defer close(terminated)
        for {
            select {
                case s: <strings:
                    // do some work
                    fmt.Println(s)
                case <-done:
                    return
            }
        }
    }()
    return terminated
}

done := make(chan interface{})
terminated := doWork(done, nil)

go func() {
    // cancel in 1 second
    time.Sleep(1 * time.Second)
    fmt.Println("canceling doWork goroutine...")
    close(done)
}()

<-terminated
fmt.Println("Done. ")

done channel 负责退出, 在goroutine的select里面作为case. 默认作为第一个参数传入.

newRandStream := func() <-chan int {
    randStream := make(chan int)
    go func() {
        defer fmt.Println("newRandStream closure exited.")
        defer close(randStream)
        for {
            randStream <- rand.Int()
        }
    }()
    return randStream
}

randStream := newRandStream()
fmt.Println("3 random ints:")
for i := 1; i<=3;i++ {
    fmt.Printf("%d: %d\n", i, <-randStream)
}

这个例子会 goroutine 泄漏(newRandStream closure exited 是否被打印可以作为是否泄露的检测标志), newRandStream 中的 goroutine 会持续等待写入 randStream 而不会退出.

解决方法跟上面一样, 增加select, 然后给一个channel作为信号.

一般创建 goroutine 的 goroutine 也负责停止 goroutine.

or-channel

var or func(channels ...<-chan interface{}) 
    <-chan interface{}
    or = func(channels ...<-chan interface{}) <-chan interface{} {
        switch len(channels) {
            case 0:
                return nil
            case 1:
                return channels[0]
        }

        orDone := make(chan interface{})
        go func() {
            defer close(orDone)

            switch len(channels) {
                case 2:
                    select {
                        case <-channels[0]:
                        case <-channels[1]:
                    }
                default:
                    select {
                        case <-channels[0]:
                        case <-channels[1]:
                        case <-channels[2]:
                        case <-or(append(channels[3:], orDone)...):
                    }
            }
        }()
        return orDone
    }

利用 or-channel 模式实现同时关闭多个 goroutine, 实际上这个实现有点像模式匹配.

错误处理

这里举了个包裹过深无法返回err的例子. 更好的方案是这样的:

type Result struct {
    Error error
    Response *http.Response
}

checkStatus := func(done <-chan interface{}, urls ...string) <-chan Result {
    results := make(chan Result)
    go func() {
        defer close(results)
        for _, url := range urls {
            var result Result
            resp, err := http.Get(url)
            result = Result{Error: err, Response: resp}
            select {
                case <-done:
                    return
                case results <- result:
            }
        }
    }()
    return results
}

done := make(chan interface{})
defer close(done)
urls := []string{"https://www.google.com", "https://badhost"}
for result := range checkStatus(done, urls...){
    if result.Error != nil {
        fmt.Printf("error: %v", result.Error)
        continue
    }
    fmt.Printf("Response: %v\n", result.Response.Status)
}

包裹错误信息, 并收集到结果中, 然后一并返回, 没什么新奇的地方.

pipline

pipline stage 的属性是:

  • 输入参数与返回值类型相同的stage
  • 一个stage必须通过编程语言进行"实化"之后才能被当作参数四处传递. Go语言中的函数就是一种"实化"并且很好地贴合我们的需求. (这应该讲的是函数无副作用)

这里去看 Designing-Data-Intensive-Applications Chapter 10, 批处理系统, 会更好, 里面详细论述了批处理系统的设计目的和优秀之处.

这里最后举的例子是, 不包含缓冲的单个函数组成的pipeline, 这不仅如书中所描述, 降低了可扩展性. 实际上, 这样做甚至会不能很好的利用CPU缓存, 造成性能低下.

构建 pipeline 的最佳实现

这里演示了一个用go模拟pipeline的例子, 其实用函数式语言会流畅许多. 函数全由 goroutine 运行, 并且将结果发送到每个函数新生成的channel (pipline临界部分).

// 用来向channel生成数据
generator := func(done <-chan interface{}, integers ...int) <-chan int{} 

// 乘法
multiply := func(done <-chan interface{}, intStream <-chan int, multiplier int,) <-chan int{}

// 加法
add := func(done <-chan interface{}, intStream <-chan int, additive int,) <-chan int{}

// 这里调用的时候比较有意思

done := make(chan interface{})
defer close(done)

intStream := generator(done, 1, 2, 3, 4)
pipline := multiply(done, add(done, multiply(done, intStream, 2), 1), 2)

for v := range pipline {
    fmt.Println(v)
}

这个例子的 goroutine 关闭具有传递性. 在任何时候停止都会传递关闭所有 pipline 的 goroutine. 这意味着, 只要有 done 的控制权, 就可以随时抢占. 即不会死锁.

扇入, 扇出

可以考虑在stage中使用 fanout 模式:

  • 不依赖之前stage计算的值
  • 运行耗时很长
fanIn := func(
    done <-chan interface{},
    channels ...<-chan interface{},
) <-chan interface{} {
    var wg sync.WaitGroup
    multiplexedStream := make(chan interface{})
    multiplex := func(c <-chan interface{}) {
        defer wg.Done()
        for i := range c{
            select {
                case <-done:
                    return
                case multiplesedStream <- i:
            }
        }
    }

    // read from all channels
    wg.Add(len(channels))
    for _, c := range channels {
        go multiplex(c)
    }

    // wait for alll reading ends
    go func() {
        wg.Wait()
        close(multiplexedStream)
    }()

    return multiplexedStream
}

即把多个channel聚合成一个channel, fan-in模式.

or-done-channel

orDone : =func(done, c <-chan interface{}) <-chan interface{} {
    valStream := make(chan interface{})
    go func() {
        defer close(valStream)
        for {
            select {
                case <-done:
                    return
                case v, ok := <-c:
                    if ok == false {
                        return
                    }
                    select {
                        case valStream <- v:
                        case <-done:
                    }
            }
        }
    }()
    return valStream
}

// usage

for val := range orDone(done, myChan)

即给channel的值加个外部包装好的过滤器, 用于辨别channel是否有效.

tee-channel

tee := func(
    done <-chan interface{},
    in <-chan interface{},
)(_, _<-chan interface{}) {
    out1 := make(chan interface{})
    out2 := make(chan interface{})
    go func() {
        defer close(out1)
        defer close(out2)
        for val := range orDone(done, in) {
            var out1, out2 = out1, out2
            for i := 0; i< 2; i++ {
                select {
                    case <-done:
                    case out1 <- val:
                        out1 = nil
                    case out2 <- val:
                        out2 = nil
                }
            }
        }
    }()
    return out1, out2
}

如同 tee 命令, 将输出流重定向. 这个源码多了一块, 估计是编辑错误. 去掉函数声明返回后面那一部分就对了.

桥接 channel

将充满 channel 的 channel 拆解为一个简单的 channel (桥接 channel)

<-chan <-chan interface{}

bridge := func(
    done <-chan interface{},
    chanStream <-chan <-chan interface{},
){
    valStream := make(chan interface{})
    go func(){
        defer close(valStream)
        for {
            var stream <-chan interface{}
            select {
                case maybeStream, ok := <-chanStream:
                    if ok == false {
                        return
                    }
                    stream = maybeStream
                case done:
                    return
            }
            for val := range orDone(done, stream) {
                select {
                    case valStream <-val:
                    case <-done:
                }
            }
        }
    }()
    return valStream
}

即, 用 orDone 将 channel 中的所有 channel 的数据取出来放到一个 channel 里面.

队列排队

队列被考虑为优化的最后手段.

这里讨论了队列的两种用途之一: 调度. 让任务的优先级尽可能相同.

队列另一个用途则是强制有序完成一致性的各种目标.

缓冲如何影响性能的例子:

    valueStream := make(chan interface{})

        defer close(valueStream)

                select {
                    case <- done:
                        return
                    case valueStream <- v:
                }
            }
        }
    }()
    return valueStream
}

    num int,

    takeStream := make(chan interface{})
    go func() {
        defer close(takeStream)
        for i := 0; i<num;i++{
            select {
                case <- done:
                    return 
                case takeStream<- <-valueStream:
            }
        }

    defer close(done)

    }
}

context 包

func main() {
    var wg sync.WaitGroup
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()
    wg.Add(1)
    go func() {
        defer wg.Done()
        if err := printGreeting(ctx); err != nil {
            fmt.Printf("cannot print greeting: %v\n", err)
            cancel()
        }
    }()
    wg.Wait()
}

func printGreeting(ctx context.Context) error {
    greeting, err := genGreeting(ctx)
    if err != nil { return err }
    fmt.Printf("%s world!\n", greeting)
    return nil
}

func printFarewell(ctx context.Context) error {
    farewell, err := genFarewell(ctx)
    if err != nil { return err }
    fmt.Printf("%s world!\n", farewell)
    return nil
}

func genGreeting(ctx context.Context) (string, error) {
    ctx, cancel := context.WithTimeout(ctx, 1*time.Second)
    defer cancel()
    switch locale, err := locate(ctx); {
        case err != nil:
            return "", err
        case locale == "EN/US":
            return "hello", nil
    }
    return "", fmt.Errorf("unsupported locale")
}

func genFarewell(ctx context.Context) (string, error) {
    switch locale, err := locate(ctx); {
        case err != nil:
            return "", err
        case locale == "EN/US":
            return "goodbye", nil
    }
    return "", fmt.Errorf("unsupported locale")
}

func locale(ctx context.Context) (string, error) {
    select {
        case <-ctx.Done():
            return "", ctx.Err()
        case <-time.After(1*time.Minute):
    }
    return "EN/US", nil
}
